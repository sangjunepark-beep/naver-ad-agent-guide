토큰 소비 절감을 위해 모델을 계층적으로 분리해서 사용하고, 컨텍스트 사용률 임계점마다 사용자에게 자동 보고한다.

**Why:** 2026-04-16 박상준 차장이 프로젝트 전반의 토큰 소비를 관리하기 위해 세팅한 규칙. Phase 1~4까지 오면서 컨텍스트 누적·반복 호출 비용이 커짐. 최종 결정권·사전 컨펌 원칙(feedback_decision_and_token_confirmation.md)의 실행 세부 규칙에 해당.

**How to apply:**

### 1) 모델 혼합 규칙
- **Haiku**: 단순 반복·기계 작업 (어드민 체크박스 토글, 시트 값 조회/붙여넣기, 파일 복사, 단순 조회, JSON 한 줄 수정)
- **Sonnet (기본값)**: 실무 대부분 (n8n 노드 편집·디버깅, 카드뉴스·리포트 작성, 상세페이지 초안, SEO 글, 일반 분석)
- **Opus**: 구조 설계·전략 판단 (Phase 설계, 파이프라인 아키텍처, 규칙 제정, 복잡한 에러 근본원인 추적)
- Haiku/Opus가 필요하면 **착수 전 "이 작업은 ○○ 모델로 돌리겠습니다" 한 줄 보고 후 진행**
- 모델 전환은 사용자가 Cowork UI에서 수동 전환하는 구조이므로 Claude는 요청 형태로 안내

### 2) 컨텍스트 사용률 알림 규칙
- **~50%**: 무알림, 정상 진행
- **50~70%**: 작업 마무리 시 "현재 컨텍스트 약 ○○% 사용 중" 한 줄 고지
- **70~85%**: 즉시 경고. "토큰 소비 급증 구간입니다. 작업 분할 or 메모리 저장 후 새 세션 권장" 컨펌 요청
- **85%+**: 작업 중단. 진행 중 산출물 요약 → 메모리 저장 → 새 세션 유도

### 3) 추가 룰
- 긴 파일(>1000줄) 읽기 전 "전체 읽으면 컨텍스트 ○○% 소모 예상" 보고 후 부분 읽기 우선 제안
- 에이전트(Task) 호출 결과가 클 것으로 예상되면 **요약 형태로 받기**
- 반복 시트 조회·n8n 워크플로우 전체 덤프는 필요한 부분만 grep/부분 읽기
- 컨텍스트 실시간 수치는 추정치이므로 사용자 쪽 Cowork UI 게이지로 교차 확인 권장
